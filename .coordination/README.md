# .coordination/ — Buddy ↔ Lead File Channel

> Convention layer for asynchronous Buddy (claude.ai chat / Claude Desktop) ↔ Lead (Claude Code) coordination. Replaces Jorge couriering between the two via copy-paste.

**Created:** 2026-05-02 as part of SG-001 Node 2.

---

## Purpose

This folder is the canonical handoff channel between Buddy (Claude in claude.ai or Desktop — the "COO") and Lead (Claude Code in terminal — the "executor"). Files here are **read by both, written primarily by Lead**, and committed to the repo so they're visible whether Buddy reads via local filesystem MCP (Desktop) or GitHub connector (web chat) — i.e., path-agnostic across Node 1 alternatives.

Goal: reduce the courier overhead that ate ~30-40% of session budget pre-SG-001. The chat channel has been lossy this project's history (~8 truncated message heads through 5/2/26). The file system has been reliable. This folder formalizes that.

---

## File inventory

| File | Writer(s) | Readers | Purpose |
|---|---|---|---|
| `decisions.md` | Lead (after Jorge/Buddy makes a call) | All | Resolved choices with reasoning + source. Append-only chronological log. |
| `questions.md` | Lead, Buddy | Jorge primarily; Buddy/Lead when answer lands | Open asks awaiting Jorge's call. Load-bearing inventions, architectural ambiguities, schema clarifications. |
| `status.md` | Lead (single writer, to avoid races) | All | Live state snapshot: current ticket, branch, teammate status, last commits, open questions, blockers, next move. Overwritten as state changes. |
| `buddy_context.md` | Lead (curated, on demand) | Buddy primarily | Session-bootstrap digest for Buddy. Tight summary of what's where, what shipped, what's pending, what files to read next. Regenerated when state shifts materially. |

Per-ticket scratch files (e.g., `MI-109-handoff.md`) MAY live in repo root rather than here (cf. existing `MI109_HANDOFF.md`); `.coordination/` is reserved for the four canonical channels above.

## Write conventions

### General

- All four files are **Git-tracked** and **committed** when state shifts. The commit IS the publication event for the GitHub-connector Node 1 path. For the local-FS-MCP Node 1 path, save-to-disk is enough — but we commit anyway for cross-device consistency and audit trail.
- All entries include an **absolute timestamp** (ISO 8601 UTC or human-readable with timezone). Relative times (`yesterday`, `tomorrow`) decay; absolutes don't.
- File channel beats chat channel for any payload >~10 lines.

### `decisions.md` (append-only)

- One decision per block. Header: `## YYYY-MM-DD HH:MM TZ — <short title>`.
- Body fields:
  - `**Decision:** <one line>`
  - `**Reasoning:** <one line — the why, not the what>`
  - `**Source:** <Jorge / Buddy / Lead-derived from X>`
  - `**Affects:** <ticket(s) or scope>`
- Never edit prior entries. If a prior decision is overturned, append a new entry with `**Supersedes:** <prior timestamp + title>`.

### `questions.md` (active queue)

- One question per block. Header: `## Q-<n>: <short subject>` (n is monotonic; never reused, even after archive).
- Body fields:
  - `**Asked by:** <name>`
  - `**Awaiting:** <name>`
  - `**Context:** <one paragraph, or pointer to file:line>`
  - `**Specific call needed:** <exact ask>`
  - `**Status:** open | answered`
- When answered, append `**Resolved:** YYYY-MM-DD HH:MM — <one-line resolution>` and flip Status to `answered`. Don't delete; the audit trail matters.
- Mirrors the load-bearing-INV protocol from MI-109's `discovery/whiteboard_override_template.md` Inventions section.

### `status.md` (overwritten)

- Single canonical state snapshot. Replace the whole file when state shifts materially.
- Required sections:
  - Header: `**Last updated:** YYYY-MM-DD HH:MM TZ` and `**Updated by:** <Lead / Jorge>`
  - `## Current ticket` — name + phase + status one-liner
  - `## Branch` — branch name + commits-ahead-of-origin
  - `## Teammates` — agent team members + their state (idle / in_progress / blocked / completed)
  - `## Last 3 commits` — `<hash> <message>`
  - `## Open questions` — Q-IDs (link to questions.md)
  - `## Blockers` — Jorge-owned external blockers
  - `## Next move` — what happens when work resumes
- Update at end of every working block; minimum at session close.

### `buddy_context.md` (curated, regenerated)

- Bootstrap document for Buddy. Keep tight: ~30-80 lines.
- Header:
  - `**Generated:** YYYY-MM-DD HH:MM TZ`
  - `**Stale after:** <date or condition — e.g., "next session-close" or "merge of PR #N">`
- Body cover: what we're working on, what's shipped (last 1-2 commits), what's pending (open questions / Phase X next), what just happened that's not yet in CLAUDE.md/STATE.md, where to look for what.
- Regenerated by Lead on demand or when state shifts materially. Stale > 24h or past the `Stale after:` condition = treat with caution; defer to STATE.md + status.md as authoritative.

## Read conventions

### Buddy (claude.ai chat or Claude Desktop)

1. **Session open:** Read `buddy_context.md` first. Cross-check `Generated:` timestamp — if stale, jump to `status.md` for the current snapshot.
2. **Decision lookup:** `decisions.md` (search by ticket / topic; entries are reverse-chronological by file position when scrolled to bottom).
3. **Pending asks:** `questions.md` filtered to `Status: open`.
4. **Source-of-truth conflicts:** `STATE.md` (repo root) > `status.md` (this folder). STATE.md is the slower-moving authoritative state; status.md is the faster-moving working snapshot. Decisions in `decisions.md` are recorded after-the-fact and don't override the conversation that produced them.

### Lead (Claude Code)

1. **Session open:** Read `status.md` first (own working state), then `questions.md` (any open Lead-owed answers).
2. **Before any major write:** Check `decisions.md` for prior calls on the same topic — don't re-decide what's already decided.
3. **Before escalating to Jorge:** Check `questions.md` — is the ask already pending there? If yes, ping Jorge in chat referencing the Q-ID rather than re-asking from scratch.

### Jorge

- Any file as needed. `questions.md` is the queue when batch-resolving open asks at session boundaries.
- Jorge does not normally edit files in this folder — Lead writes; Jorge confirms or redirects via chat. (Jorge MAY edit directly if a quick correction is faster than briefing Lead.)

## Lifecycle

- **Created on first use per project.** The first ticket that needs the channel triggers `git add .coordination/<file>.md` + commit.
- **Per-project, not per-ticket.** Same `.coordination/` serves the whole project; tickets are referenced *inside* files, not as separate folders.
- **Cleanup:** at v0.1 / v1.0 milestones, archive old entries to `.coordination/archive/<YYYY-MM>/` to keep the working files lean. Append-only files (`decisions`, `questions`) get pruned this way; `status` and `buddy_context` are already overwritten in place.

---

## Conflict rules

- `STATE.md` (repo root) **>** `status.md` (this folder). STATE.md is authoritative for slow-moving state; status.md is the working snapshot.
- `CLAUDE.md` (repo root) **>** `decisions.md` (this folder) for locked principles. `decisions.md` captures the *path* to the principle; CLAUDE.md captures the *principle itself*. When a decision graduates to a locked principle, bank it into CLAUDE.md and let the `decisions.md` entry reference forward.
- Chat memory **<** any file in this folder. Memory drifts; files don't.
