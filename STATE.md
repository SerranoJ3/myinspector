### Lesson 7 — Verify RLS WITH CHECK columns are populated on every INSERT

**Rule:** before shipping any client-side INSERT to a table with RLS, query `pg_policies` for the WITH CHECK expression and verify every column referenced is populated by the client code. RLS rejection on missing WITH CHECK columns is silent at the SQL level — Supabase returns success-shaped responses for inserts that policy actually rejected, and the row never lands. UI doesn't surface the failure unless the code explicitly inspects `error` on the insert response.

**Why:** Luis v1 chat panel was deployed with `luis_conversations.insert({ user_id, project_id, module_key, question, answer, sources })` — missing `firm_id`. `pg_policies` showed the WITH CHECK as `(firm_id = current_firm_id()) OR is_super_admin()`. NULL firm_id failed the check, but the client treated the response as success because no thrown error surfaced. **Every Luis conversation written before the Thu 5/7 ~00:25 EDT fix was silently dropped.** Unknown how long the bug had been live; whoever was using Luis on prod (Jorge during demo prep) saw the chat work in-session but rows never persisted.

**How to apply:** at any client-side INSERT against an RLS-protected table:
1. Query `pg_policies` for the WITH CHECK expression on that table
2. Cross-reference every column the policy references (typically `firm_id`, sometimes `submitted_by`, `actor_id`, `created_by`)
3. Verify the client INSERT payload includes ALL of them, populated from authenticated session context (`currentFirmId`, `currentInspectorId`, etc.)
4. After deploy, run a ground-truth row-count check (`SELECT COUNT(*) FROM <table> WHERE created_at > <deploy_time>`) to confirm rows are actually landing, not just the API returning success-shape.

**Banked rationale:** RLS is fail-closed by design (rejected rows don't leak to other firms), but it's silent-fail when the client doesn't inspect errors. The cost of a missing column is "feature appears to work, no data persisted" — exactly the worst failure mode for a demo or production user. The 4-step check above takes ~2 minutes and prevents the whole class of bug.

**Counter-cases:** server-side INSERTs running as `service_role` bypass RLS entirely, so this lesson doesn't apply (different attack surface). Apply only to client-side / user-JWT-context INSERTs.

### Lesson 8 — Honor schema-level identity-display flags at every render site, not just write paths

**Rule:** when a sensitive table has an "is X safe to surface" boolean flag (e.g., `firms.firm_safe_to_display`, future `person_safe_to_display`), every UI render site that displays a value from that row must check the flag before rendering. The flag is inert if no display site enforces it. Pitch-mode-style write-suppression is necessary but not sufficient — read-side display redaction is its own enforcement locus, orthogonal to write suppression.

**Why:** Thu 5/7 evening, Jorge's click-test screenshot on the `demo-banner` Vercel preview alias showed the sidebar profile card displaying "CP Engineers" as the subtitle under "Jorge Serrano" — a clean identifying-material leak even though pitch mode was active. Pitch mode suppresses writes, not display. The `firms.firm_safe_to_display` boolean had been added at a prior point and the column was already loaded by `initApp` (queried for the demo banner gate), but the actual `.user-role` display element wasn't gating on it before rendering the firm name. Both data and access lived behind RLS-correct paths; the failure was purely on the render-site discipline. MI-DEMO-UI v3 (`ec1f981`, +19/-3 lines, 5 surgical Lead Edits) wired the gate. Buddy's pre-work-order analysis captured the orthogonality framing that distinguishes this from the existing pitch-mode regime: pitch mode = write-suppression toggle (operator-controlled); firm_safe_to_display = display-redaction flag (schema-controlled). All four combinations valid (off/off, off/on, on/off, on/on) — separate variables, separate gates.

**How to apply:** any new column on a sensitive table that's a "is X safe to surface" flag has two enforcement loci:
1. **Backend (RLS):** can the user query the row at all? (Existing discipline.)
2. **Frontend (display gate):** even if the user CAN query it, should the value render to the current display surface? (New discipline locked by Lesson 8.)

When wiring such a flag:
- Capture it in the same `initApp`-style profile-fetch path that pulls the row (no separate query).
- Default the runtime variable to the safe value (`false` for "is safe to display") so pre-login + post-logout chrome stays redacted regardless of caching state.
- Reset the variable on `logout` for the same reason.
- Grep for every render site that would surface the value before claiming the gate is complete (MI-DEMO-UI v3 grep found 2 sites — `.user-role` sidebar + signup toast — and gated both).
- For any render site where the flag isn't fetchable in time (e.g., the signup toast where the `lookup_firm_by_code` RPC return shape doesn't include the flag and anon access to `firms` was dropped), default-redact (generic copy) is the safe baseline. Don't surface the value just because gating it is inconvenient.

**Counter-cases:** super_admin badges and role indicators are independent of identity flags — they describe the user's access level, not the firm's identity ratification status. Don't gate them on identity flags. Same for badges that describe system state ("Pitch mode ON", "Demo session", etc.) — those describe the session, not the firm.

**Banked rationale:** the cost of a missing display gate is "real customer's name appears in a prospect-facing demo screen-share" — silent, embarrassing, and can torpedo a sales conversation. The cost of adding the gate is one boolean check per render site. The asymmetry justifies the discipline. Future `person_safe_to_display`, `project_safe_to_display`, etc. patterns should follow the same Lesson-8 enforcement: every render site checks the flag, default-safe variables on session boundaries, grep at ship time to verify coverage.

**Counter-cases:** when the directive is explicit about element type AND ID/class (e.g., "the `<button id='submit-tapcard'>` button"), grep for the identifier instead. When the directive is about adding a new element (no existing element to find), grep is unnecessary. Apply Lesson 9 specifically to "modify/fold/move/remove the [labeled element]" directives.

**Banked rationale:** ~30 seconds of grep prevents directives-from-memory from producing wrong builds. The asymmetry is even sharper than Lesson 1 (stop-and-ping when brief contradicts codebase) because remembered labels are higher-confidence-feeling than briefs ("I just looked at it") but actually more error-prone (memory-of-screenshot vs current-state-of-tree drift). The grep is cheap; the build-fix-revert cycle isn't.

### Lesson 11 — Orphan tapcard pattern: design read-side fallbacks for write-side state that can't be guaranteed

**Rule:** when a table has a nullable FK that *should* be populated but isn't always (because the producer flow can't always populate it), build the read path with a fallback rather than treating the FK as authoritative. The right architecture is to fix the producer to always populate; in the meantime, the consumer needs a fallback or the data appears missing.

**Why:** every drawn tapcard in production has `phase_submissions.materials_sheet_id: null` because `currentTapcard.materials_sheet_id = null` at submit time (the tapcard was submitted before any materials_sheet existed for the property — a common workflow where inspectors draw the diagram first and fill the MS later). The original `vtcLoadTapcardData(materialsSheetId)` query was strict: if the FK was null, the function returned null and the VTC paper preview rendered an empty diagram. **Result: 100% of in-the-wild tapcards rendered as empty scaffolds when viewed from the MS modal**, even though the data was sitting in `phase_submissions.tapcard_data.diagram`. Read-side fallback `vtcLoadTapcardData(materialsSheetId, propertyIdFallback)` (commit `c21a7da` Sun 5/10) retries by `property_id` when the materials_sheet_id query returns null. Write-side architecture fix (auto-link at submit time when MS exists for property) parked POST-DEMO since the read-side fallback patches the display path and the write-side fix has broader implications.

**How to apply:** for any nullable FK on a high-traffic read surface:
1. Audit: how often is the FK actually populated in production? Run `SELECT COUNT(*) WHERE fk IS NULL / TOTAL` — if non-trivial percentage, the producer is not always populating it.
2. If the producer can be fixed: file a ticket for the write-side fix.
3. Regardless of producer-fix timeline: build the read path with a fallback. The fallback should use whichever column IS reliably populated (typically a different FK, a UUID embedded in a JSONB column, or a higher-level entity ID) to surface the right row.
4. Document the fallback as a known pattern in the consumer function's docstring + STATE.md (so future readers don't see "redundant FK" and try to clean it up).

**Counter-cases:** when the FK is reliably populated by the producer (verified via row-count audit), strict FK enforcement is correct — fallbacks add complexity and can mask real producer bugs. Apply Lesson 11 only when the producer flow can't always populate the FK and the data is critical to display.

**Banked rationale:** the cost of strict FK enforcement when production data has gaps is "feature looks broken to every user with affected rows" — exactly the demo-killer failure mode. The cost of a fallback is one extra query branch + a sync note. Asymmetry favors the fallback. Same pattern would apply to: `phase_submissions.parent_phase_id` (chain reconstruction), `daily_reports.generated_by` (if user account deleted but report still relevant), etc. Build read-side fallbacks for any nullable FK that displays data the user expects to see.

### Lesson 12 — Visual extent vs center: clamp the visible rect, not the geometric center

**Rule:** when clamping draggable coordinates inside a container, account for the visual extent of the rendered element (rect dimensions + selection ring + label overflow) — not just the geometric center. Clamping the center to `[0, 1]` puts visible elements past the edge of the container by half their visible width.

**Why:** Mon 5/12 ~afternoon Jorge reported "asset overflows the canvas on 124 Oak Street." The asset center was at (0.97, 0.5) — within the original `[0, 1]` clamp — but the asset itself was a 24x24 rect (12px half-width) plus a 22px-radius selection ring when selected, so the visible extent reached past x=1.0. Fix (commit `aecc952`): introduce `DIAGRAM_DRAG_INSET = 0.03` and clamp drag coords to `[0.03, 0.97]`. 0.03 normalized = 24px in 800w viewBox / 18px in 600h viewBox, which covers the widest visual element (the selection ring at r=22). Side benefit: auto-computed distance/bearing reads from the clamped position, so measurements automatically reflect what the inspector physically placed within bounds — matches Jorge's design intent without a separate code path.

**How to apply:** for any drag-clamp boundary in an SVG / canvas / coordinate-space-based editor:
1. Identify the widest visual element rendered at that coordinate (rect, ring, label, badge). Don't assume the visible width equals the rect width — selection rings, drop shadows, and labels can extend further.
2. Compute the half-width in normalized coords: `half_width / viewBox_width`.
3. Set the inset to that half-width (or slightly larger for breathing room). Clamp to `[inset, 1 - inset]` on both x and y.
4. If the element scales with selection state (rect vs rect+ring), use the larger state's extent for the inset.

**Counter-cases:** when the visible element IS the center point (e.g., a 1px dot, a pure-text label anchored to center), `[0, 1]` clamp is correct because the visible extent equals the center. Apply Lesson 12 specifically to extended elements (rects, circles with radius > 1, multi-pixel markers).

**Banked rationale:** the cost of clamping the wrong reference (center) is "elements look like they overflow when they shouldn't" — visible UX bug. The cost of clamping the right reference (visible extent) is one constant + one subtraction per clamp call. Trivial to get right at design time; expensive to retrofit when users start reporting visible overflow.

### Lesson 15 — Query pg_proc for canonical function names; don't assume Postgres convention applies

**Rule:** before referencing a Postgres function in a migration or apply, query `pg_proc` for the actual function definition by name. MyInspector has legacy naming conventions where canonical helpers (especially the updated_at trigger function) live under topic-specific names rather than the Postgres-convention `update_updated_at_column`. Assuming convention applies will produce migrations that reference a non-existent function and fail silently in some Postgres versions or break dependent triggers in others.

**Why:** Wed 5/13 evening Buddy's first attempt at the OPS Dashboard schema migration referenced `update_updated_at_column()` for the updated_at triggers on the 4 new tables. Migration applied successfully but the triggers failed at runtime because `update_updated_at_column` only exists in the `storage` schema (Supabase's own internal helper for storage.objects). The MyInspector public schema's canonical updated_at function is `gis_set_updated_at()` — named after MI-401 GIS lists where it originated, then reused across firms / heralds / phase_submissions / gis_lists / gis_list_entries. The naming is non-obvious; without querying `pg_proc` first, the wrong function name would have shipped silently and the triggers would have never fired.

**How to apply:** before referencing any Postgres helper function in a migration:
1. `SELECT proname, pronamespace::regnamespace FROM pg_proc WHERE proname ILIKE '%<keyword>%'` (or a slightly broader pattern if the topic is unclear).
2. Identify all candidates in the `public` schema (not `storage`, `auth`, `extensions`, etc.).
3. If there's a topic-specific legacy name (e.g., `gis_set_updated_at` for updated_at, `audit_log_<table>` for audit triggers), use that.
4. If there's no existing helper, write a new one named per the established pattern (topic + verb + target) — don't introduce the Postgres-convention name and create a new naming locus.
5. Verify the chosen function exists in the right schema before referencing it in CREATE TRIGGER.

**Counter-cases:** Supabase's own internal helpers (`storage.update_updated_at_column`, `auth.*`) are fair game when working in those schemas. Apply Lesson 15 specifically to public-schema migrations referencing public-schema helpers.

**Banked rationale:** Lesson 10 deepening (verify reality before acting on assumed convention). The cost of `SELECT proname FROM pg_proc` is one MCP call; the cost of a migration with a wrong function reference is silent trigger failure that surfaces only when a downstream feature breaks. Asymmetry favors verification. Standing rule: pg_proc is the truth about what functions exist; STATE.md / decisions.md / brief docs are at best partial mirrors and can drift.

### Lesson 16 — Postgres CTE multi-write against the same row fails silently; use sequential statements or CASE

**Rule:** when bulk-updating multiple columns on the same row via a single SQL statement, do NOT use multiple chained CTEs with separate UPDATE clauses targeting the same row. Postgres serializes CTE writes within a single statement and silently drops all but one write when the targets overlap. Use sequential UPDATE statements OR a single UPDATE with a `CASE` expression covering all the columns in one pass.

**Why:** Wed 5/13 evening Buddy's demo photo replacement migration used a multi-CTE pattern: `WITH curbstop AS (UPDATE ... WHERE label='curbstop_meterpit' ...), watermain AS (UPDATE ... WHERE label='watermain_corporation' ...), ...`. Expected 49 photos updated; actual `RETURNING` count showed 36 updated. The dropped 13 photos were exactly the rows where the SAME phase_submission had both curbstop AND watermain photo entries — the multi-CTE pattern only applied one of the two updates per row. Required 3 follow-up UPDATE statements (sequential, scoped to specific labels + submission types) to backfill the 13 missing rows. Discovered when a `SELECT COUNT(*) WHERE photo_url LIKE '%placehold.co%'` audit ran post-migration and surfaced 13 stale URLs that the migration claimed to have replaced.

**How to apply:** when bulk-touching multiple columns or multiple values on the same row:
1. **Preferred (single statement, atomic):** use a single UPDATE with `CASE WHEN <condition> THEN <value> ELSE <fallback> END` for each column being updated. One pass, no overlap issue, fully atomic.
2. **Alternative (sequential, still safe):** use N separate UPDATE statements wrapped in a transaction (`BEGIN; UPDATE ...; UPDATE ...; COMMIT;`). Each statement sees the prior statement's writes; final state is correct.
3. **Avoid:** multi-CTE pattern with separate UPDATE clauses targeting overlapping rows. The pattern LOOKS atomic but silently drops writes.
4. **Always audit post-migration** with a query that confirms the expected count of updated rows actually changed (`SELECT COUNT(*) WHERE <expected_post_state>`). Don't trust `RETURNING` row count from the migration itself — it reports total CTE actions, not unique rows successfully updated.

**Counter-cases:** CTEs that read-then-write to DIFFERENT tables (e.g., `WITH inserted_parent AS (INSERT INTO parent RETURNING id), child AS (INSERT INTO child SELECT id FROM inserted_parent)`) are fine — the silent-drop only happens when multiple WRITE CTEs target the same rows in the same table. Apply Lesson 16 specifically to overlapping multi-write CTEs.

**Banked rationale:** the cost of using sequential statements or CASE is verbose SQL (2-5 extra lines); the cost of a silent-drop CTE is data integrity failure that doesn't surface until a downstream audit catches it (or worse, surfaces during a demo when the row should have been updated but wasn't). Asymmetry favors verbose-but-correct. Standing rule: any time the same row appears in 2+ WHERE clauses of the same migration, switch to CASE or sequential. The Postgres docs (`https://www.postgresql.org/docs/current/queries-with.html` §7.8.2 "Data-Modifying Statements in WITH") explicitly call out this hazard but it's easy to miss when scanning for syntax rather than semantics.
